## 1. GitHub 활동 자동 피드 (Auto Activity Feed)

GitHub Events API가 이미 sync-activity에서 사용 중. 이를 확장하여 **자동 피드**로 만든다.

### DB: `activity_feed` 테이블 (migration 014)
```sql
CREATE TABLE activity_feed (
  id TEXT PRIMARY KEY,
  actor_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  event_type TEXT NOT NULL,  -- 'github_push', 'github_pr', 'github_star', 'github_release', 'follow', 'star_post', 'fork_post', 'reply'
  target_user_id TEXT REFERENCES users(id),
  target_post_id TEXT REFERENCES posts(id),
  metadata TEXT DEFAULT '{}',  -- JSON: repo, branch, commits count, PR title, etc.
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_activity_actor ON activity_feed(actor_id, created_at DESC);
CREATE INDEX idx_activity_created ON activity_feed(created_at DESC);
```

### Server: `GET /api/activity/feed`
- Returns activity from users you follow + your own
- Cursor-based pagination
- Merges GitHub events (from Events API polling) + internal Forkverse events

### Server: `POST /api/activity/sync-github`
- Called periodically (or on page load)
- Fetches `/users/{username}/events` from GitHub API
- Converts PushEvent, PullRequestEvent, WatchEvent, ForkEvent, ReleaseEvent, CreateEvent → activity_feed rows
- Deduplicates by event ID (using github_synced_events table)

### Client: ActivityFeedPage (`/activity`)
- Terminal-style activity stream
- Each event rendered as CLI command:
  ```
  ■ @alice git push --repo=alice/app --commits=3           2m ago
    "드디어 auth 버그 잡았다"                              ← optional comment
    🚀 5  lgtm 3

  ■ @bob gh pr merge #42 --repo=team/api                  1h ago
    💬 2  🔥 4

  ■ @charlie mass-star 4 repos                            3h ago
    tailwindcss/tailwindcss, shadcn/ui, ...
  ```

### Internal events auto-created:
- When user follows → `activity_feed` row (event_type: 'follow')
- When user stars a post → row (event_type: 'star_post')
- When user forks a post → row (event_type: 'fork_post')
- When user replies → row (event_type: 'reply')

## 2. 알림 시스템 (Notifications)

### DB: `notifications` 테이블 (migration 015)
```sql
CREATE TABLE notifications (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  type TEXT NOT NULL,  -- 'star', 'reply', 'follow', 'mention', 'fork', 'github_pr_review'
  actor_id TEXT NOT NULL REFERENCES users(id),
  post_id TEXT REFERENCES posts(id),
  message TEXT,
  read INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_notifications_user ON notifications(user_id, read, created_at DESC);
```

### Server endpoints:
- `GET /api/notifications` — list notifications (unread first, cursor pagination)
- `GET /api/notifications/unread-count` — returns `{ count: N }` (for header badge)
- `POST /api/notifications/:id/read` — mark one as read
- `POST /api/notifications/read-all` — mark all as read

### Server: auto-create notifications
- In `POST /posts/:id/star` → create notification for post author
- In `POST /posts` (reply) → create notification for parent author + mentioned users
- In `POST /users/@:username/follow` → create notification for target user
- In `POST /posts/:id/fork` → create notification for original post author

### Client: NotificationBell (HeaderBar component)
- Bell icon in header with unread count badge
- Dropdown panel showing recent notifications:
  ```
  $ notifications --unread
  > @alice ★ starred your post        2m ago
  > @bob ↵ replied to your post       5m ago
  > @charlie → followed you           1h ago
  > @dave @ mentioned you             3h ago
  ```
- Click notification → navigate to relevant post/profile
- "mark all read" button

## 3. 터미널 리액션 (Reactions)

GitHub style reactions but terminal-themed.

### DB: `reactions` 테이블 (migration 016)
```sql
CREATE TABLE reactions (
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  post_id TEXT NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  emoji TEXT NOT NULL,  -- 'lgtm', 'ship_it', 'fire', 'bug', 'thinking', 'rocket', 'eyes', 'heart'
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (user_id, post_id, emoji)
);
CREATE INDEX idx_reactions_post ON reactions(post_id);
```

### Emoji set (terminal-themed display):
| key | display | meaning |
|-----|---------|---------|
| lgtm | `[lgtm]` | Looks good to me |
| ship_it | `[ship]` | Ship it! |
| fire | `[fire]` | This is fire |
| bug | `[bug]` | Found a bug |
| thinking | `[hmm]` | Thinking... |
| rocket | `[rocket]` | To the moon |
| eyes | `[eyes]` | Looking at this |
| heart | `[heart]` | Love it |

### Server:
- `POST /api/posts/:id/react` — body: `{ emoji: string }` (toggle)
- `GET` response on posts includes `reactions: { lgtm: 3, fire: 1, ... }` and `myReactions: ['lgtm']`

### Client: ReactionBar below ActionBar
```
[lgtm] 3  [fire] 1  [rocket] 2  [+]
```
- Click `[+]` opens reaction picker
- Click existing reaction toggles your reaction on/off

## 4. 유저 추천 (Who to Follow)

### Server: `GET /api/users/suggested`
Algorithm (no ML needed, pure SQL + GitHub data):
1. GitHub mutual followers: users on Forkverse who follow you on GitHub but you don't follow on Forkverse
2. Same language: users whose top_languages overlap with yours
3. Same starred repos: users who starred the same repos (via repo_attachments)
- Returns top 5, excludes already-following

### Client: SuggestedUsers component (Sidebar or Explore page)
```
// who to follow
  @alice  TypeScript · 3 mutual follows  [follow]
  @bob    Rust · starred same repos      [follow]
```

## 5. 검색 (Search)

### DB: FTS5 virtual table (migration 017)
```sql
CREATE VIRTUAL TABLE posts_fts USING fts5(
  message_raw,
  tags,
  content=posts,
  content_rowid=rowid
);
-- Triggers to keep FTS in sync
```

### Server:
- `GET /api/search?q=...&type=posts|users|tags`
- Posts: FTS5 full-text search
- Users: LIKE query on username, display_name, github_username
- Tags: prefix match on tags

### Client: SearchPage or search bar in header
```
$ search "react hooks"
> 12 posts  |  3 users  |  5 tags
```
