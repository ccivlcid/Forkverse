## 2. 최종 결론

Forkverse는 다음 방향으로 가는 것이 가장 적절하다.

### 최종 제품 방향
**GitHub 저장소를 AI로 분석하고, 그 결과를 공유 가능한 개발자 인사이트로 바꿔주는 플랫폼**

### 최종 운영 방식
**중앙 DB와 공식 도메인을 가진 SaaS 형태**

### 최종 공개 전략
**오픈코어(Open Core) + 공식 호스팅 서비스 운영**

### UX 우선순위
**Analyze first, social second**

즉, Forkverse는 단순한 개발자 SNS도 아니고, 단순한 로컬 분석 툴도 아니다.  
핵심은 **Repo Analysis Platform**이고, 소셜과 CLI 감성은 그 위에 얹히는 구조다.

## 3. 왜 B안이 맞는가

초기에는 다음 두 가지 방향이 가능했다.

### A안
개발자 소셜 네트워크 중심  
- 포스트
- CLI 변환
- 피드
- 스타/포크
- GitHub 문화 기반 소셜 인터랙션

### B안
Repo 분석 중심 플랫폼  
- GitHub repo 분석
- 분석 결과 리포트
- 결과 공유
- 분석 히스토리
- 팀/커뮤니티 재활용

최종적으로는 **B안이 더 적합하다**고 판단했다.

### 이유 1. 실용성이 더 강하다
“자연어를 CLI 스타일로 보여주는 SNS”는 재미있지만, 지속 사용 이유가 약할 수 있다.  
반면 “저장소를 분석해 바로 쓸 수 있는 인사이트를 주는 도구”는 명확한 실용성이 있다.

### 이유 2. 재방문 가치가 높다
Repo 분석 결과는 다음과 같은 방식으로 반복 사용될 수 있다.

- 새로운 오픈소스 탐색
- 팀 내 코드 이해
- 발표자료 초안 생성
- 기술 검토 자료 작성
- 인수인계/온보딩 자료 활용

### 이유 3. 수익화가 더 자연스럽다
분석 요청량, 고급 모델, private repo 분석, 팀 협업 기능 등으로 유료화 구조가 자연스럽게 이어진다.

### 이유 4. 소셜 기능도 자연스럽게 살릴 수 있다
소셜이 없어지는 것이 아니라, 다음처럼 보조 레이어로 바뀐다.

- 분석 결과 공유
- 스타
- 포크
- 재분석
- 인기 분석 피드
- 프로필 기반 활동 기록

즉, 소셜은 메인이 아니라 **분석 결과 유통 구조**가 된다.

## 5. 핵심 가치 제안

Forkverse의 핵심 가치는 다음 네 가지다.

### 1) 저장소를 빠르게 이해하게 해준다
처음 보는 repo도 빠르게 핵심을 파악할 수 있어야 한다.

### 2) 분석 결과를 재사용 가능한 산출물로 바꿔준다
단순 요약이 아니라 실제 재사용 가능한 형태여야 한다.

예:
- 리포트
- CLI 요약
- 발표자료 초안
- 공유 포스트

### 3) 개발자 문화에 맞는 표현 방식을 제공한다
CLI 감성, GitHub 문화, 개발자 지향 UX를 유지한다.

### 4) 분석 결과가 서비스 안에 축적된다
중앙 서비스에 분석 기록과 공유 결과가 쌓이면서 플랫폼 가치가 커진다.

## 6. 최종 제품 구조

Forkverse는 크게 아래 3층으로 이해하면 된다.

### 6.1 Analyze Layer
제품의 핵심 진입점이다.

주요 기능:
- GitHub repo 입력
- 분석 요청
- 모델 선택
- 분석 타입 선택
- 진행 상태 확인
- 결과 확인
- export / share

### 6.2 Insight Layer
분석 결과를 읽고 재활용하는 층이다.

주요 기능:
- Executive Summary
- Tech Stack
- Architecture Guess
- Strengths / Risks
- Suggested Improvements
- CLI View
- 결과 저장 / 이력

### 6.3 Social Layer
결과를 퍼뜨리고 재사용하는 층이다.

주요 기능:
- Feed
- Share Post
- Star
- Fork
- Profile
- Explore
- 인기 분석 결과 탐색

즉, 전체 우선순위는 다음과 같다.

**Analyze → Insight → Social**

## 7. 운영 방식 최종 결정

### 7.1 중앙 서비스형으로 간다

추천 운영 형태는 명확하다.

- 자체 도메인
- 중앙 DB
- 공식 호스팅 서비스
- GitHub OAuth 로그인
- 사용자별 분석 이력
- 공유 가능한 결과 URL
- 소셜 피드

이 방향은 단순 오픈소스 배포 방식과 다르다.

### 7.2 왜 로컬 전용 오픈소스가 아닌가

로컬 툴 중심 구조는 다음 한계가 있다.

- 분석 결과가 중앙에 쌓이지 않음
- 링크 공유 가치가 약함
- 피드가 형성되지 않음
- 플랫폼 네트워크 효과가 생기지 않음
- B안의 핵심 가치가 약해짐

즉, 로컬 툴은 “도구”는 될 수 있어도 “플랫폼”이 되기는 어렵다.

### 7.3 왜 중앙 DB가 필요한가

B안은 분석 결과를 축적하고 연결해야 가치가 커진다.

중앙 DB를 두면 가능해지는 것:
- 분석 결과 저장
- 사용자 이력 관리
- 인기 결과 집계
- 프로필 기반 활동 노출
- 공유 포스트와 원본 분석 연결
- 재분석 이력 추적
- 추후 추천 시스템 연결

## 8. 공개 전략 최종 결정

### 8.1 최종 추천 방식
**오픈코어(Open Core) + 공식 SaaS 운영**

즉,
- 제품의 중심 가치는 공식 서비스에 둔다
- 일부 코드는 GitHub에 공개한다
- GitHub는 신뢰/유입/브랜딩 채널로 사용한다

---

## 9. 오픈코어 구조

### 9.1 공식 서비스에 남길 것
비공개 또는 공식 서비스 전용으로 유지할 것

- 중앙 서비스 백엔드 핵심 로직
- 분석 파이프라인 운영 로직
- 작업 큐/워커 운영 구조
- 결과 저장 및 랭킹/추천 로직
- 사용량 제한/과금 정책
- 관리자 기능
- 운영용 프롬프트 및 품질 관리 체계

### 9.2 공개 가능한 것
개발자 신뢰와 유입을 위한 공개 대상

- 제품 소개용 public repository
- 일부 프론트 구조
- SDK / API client 일부
- CLI 또는 lite tool
- self-host 예시 버전
- 예시 분석 결과
- 아키텍처 문서 일부
- 로드맵 및 공개 문서

---

## 10. 왜 오픈코어가 적절한가

### 10.1 플랫폼 가치를 보호할 수 있다
핵심 가치가 공식 서비스 안에 남는다.

### 10.2 개발자 신뢰를 얻을 수 있다
GitHub에 아무것도 없는 상태보다 훨씬 낫다.

### 10.3 유입 경로를 만들 수 있다
GitHub 저장소 자체가 랜딩 페이지이자 신뢰 채널 역할을 한다.

### 10.4 향후 수익화가 가능하다
완전 풀오픈소스보다 수익화 설계가 쉽다.

## 12. 추천 공개 메시지

### 가장 추천하는 포지셔닝 문구
**AI가 GitHub repo를 분석하고, 그 결과를 공유 가능한 개발자 인사이트로 바꿔주는 플랫폼**

### 대체 문구
- Analyze a GitHub repo and turn it into structured developer insights
- Repo analysis with shareable reports, CLI views, and developer-first UX
- GitHub repository analysis built for developers, not generic AI summaries

### 추천하지 않는 문구
- 터미널풍 개발자 SNS
- 자연어를 CLI로 바꿔주는 소셜 플랫폼

이런 표현은 재미는 있지만 제품 효용 전달력이 약하다.

## 13. 타겟 사용자

### 13.1 핵심 타겟

#### 1) 개발자 / 테크 리드
- 처음 보는 repo를 빠르게 이해하고 싶다
- 기술 검토 자료가 필요하다

#### 2) AI/개발 도구 애호가
- 새로운 개발자용 AI 도구를 적극적으로 체험한다
- 분석 결과를 공유하는 문화에 익숙하다

#### 3) 기술 콘텐츠 제작자
- 오픈소스 소개 글이나 발표자료를 빠르게 만들고 싶다

#### 4) PM / 아키텍트 / 리크루터
- 저장소 수준과 구조를 빠르게 파악하고 싶다

## 18. Part 1 요약

Forkverse의 최종 방향은 다음과 같다.

- **제품 중심축**: Repo Analysis Platform
- **운영 방식**: 중앙 DB 기반 공식 SaaS
- **공개 전략**: 오픈코어
- **UX 우선순위**: Analyze first, social second
- **핵심 가치**: 분석 결과 생성 + 공유 + 재활용

이 기준을 흔들리지 않는 상위 전략으로 고정하고, 이후 아키텍처와 UI/UX는 모두 여기에 맞춰 정렬해야 한다.
