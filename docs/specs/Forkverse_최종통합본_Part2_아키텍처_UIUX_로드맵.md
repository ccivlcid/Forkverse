## 1. 아키텍처 총평

현재 Forkverse의 구조는 빠른 MVP 실험에는 적합하다.  
특히 다음 강점이 있다.

- 단순하고 이해하기 쉬운 개발 구조
- 문서 중심 설계
- AI 협업에 유리한 규칙 기반 프로젝트 운영
- 여러 LLM 제공자를 연결할 수 있는 유연성

하지만 장기적으로는 다음 전환이 필요하다.

- 단일 API 서버 중심 구조 → API + Worker 분리
- 동기 실행 중심 → Job 기반 비동기 처리
- SQLite 중심 운영 → 확장 가능한 데이터 구조
- 단순 멀티 모델 연결 → 운영 가능한 LLM Gateway 구조
- 기능 나열 구조 → 도메인 중심 설계

즉, 지금은 “좋은 출발점”이지만, 실제 서비스 운영형 구조로 가려면 실행 구조를 재설계해야 한다.

## 2. 핵심 구조 문제

### 2.1 API 서버에 책임이 과도하게 몰릴 가능성

현재 개념상 서버는 다음 역할을 동시에 맡을 수 있다.

- 인증
- 포스트 CRUD
- 피드 조회
- 사용자 관계 처리
- LLM 호출
- 저장소 분석
- 결과 생성
- 상태 관리
- 히스토리 저장

이렇게 되면 서버가 단순 API 서버가 아니라, 무거운 분석 실행기까지 맡게 된다.

### 문제점
- 긴 요청과 짧은 요청이 섞인다
- 분석 요청이 몰리면 일반 기능도 느려진다
- 실패 재시도 구조가 불안정해진다
- 상태 추적이 어려워진다
- 진행 상황 UX 구현이 힘들어진다

---

### 2.2 저장소 분석 기능은 본질적으로 Job 기반이다

Repo 분석은 일반 CRUD 요청과 다르다.

권장 흐름:
1. 분석 요청 생성
2. Job 적재
3. Worker 실행
4. 상태 갱신
5. 결과 저장
6. 프론트에서 상태 조회 및 표시

즉, 분석 기능은 원래부터 **비동기 작업 관리 구조**가 필요하다.

---

### 2.3 SQLite는 MVP에는 적합하지만 장기 운영에는 한계가 있다

SQLite는 초기에 빠르고 단순하다는 장점이 있다.  
그러나 아래 조건이 늘어나면 한계가 빨리 보일 수 있다.

- 동시 사용자 증가
- 분석 이력 누적
- 상태 업데이트 증가
- 피드/활동 로그 증가
- 검색 기능 강화
- 다중 인스턴스 운영

즉, MVP에는 적합하지만 장기 운영을 전제로 한 핵심 DB로 계속 가져가기에는 무리가 생길 수 있다.

---

### 2.4 멀티 LLM 구조는 운영 표준화가 필요하다

멀티 프로바이더를 지원하는 것 자체는 강점이다.  
하지만 운영 관점에서는 다음 문제가 생긴다.

- 모델별 응답 포맷 차이
- 비용 차이
- 시간 차이
- 품질 편차
- 타임아웃/에러 처리 차이

따라서 단순히 여러 모델을 연결하는 것이 아니라, 내부적으로 표준화 계층이 필요하다.

## 3. 권장 아키텍처 방향

## 3.1 3계층 구조

### 1) Web App Layer
역할:
- UI
- 분석 요청 입력
- 결과 화면
- 피드/프로필/공유 UX

기술 예시:
- React + Vite

### 2) API Layer
역할:
- 인증
- 사용자/피드/포스트/프로필 API
- 분석 Job 생성
- 상태 조회
- 결과 조회
- 권한/검증/감사

기술 예시:
- Express 또는 Fastify

### 3) Worker Layer
역할:
- GitHub repo 수집
- 코드/구조 분석
- LLM 호출
- 리포트 생성
- PPTX/추후 산출물 생성
- 실패 재시도

즉, API와 무거운 분석 실행을 분리해야 한다.

---

## 3.2 Job Queue 도입

분석 기능이 핵심이면 큐는 거의 필수다.

### 권장 흐름
1. 사용자가 분석 요청
2. API가 `analysis_job` 생성
3. 큐에 적재
4. Worker가 처리
5. 상태 업데이트
6. 완료 시 결과 저장
7. 프론트는 polling 또는 SSE로 표시

### 기대 효과
- 긴 작업과 일반 API 분리
- 상태 추적 가능
- 재시도 가능
- 작업 이력 관리 가능
- 추후 병렬 처리 가능

---

## 3.3 데이터 전략

### 초기
- SQLite 유지 가능
- 단, MVP 전용이라는 전제가 필요

### 확장
- Postgres 전환 권장
- 분석 이력, 활동 로그, 통계, 검색 확장에 유리

### 보조 구성
- Redis: 캐시 / 큐 보조 / 상태 관리
- Object Storage: PPTX, 영상, export 산출물 저장

---

## 3.4 LLM Gateway 계층

### 목적
멀티 LLM 환경을 운영 가능한 구조로 만든다.

### 공통 기능 예시
- `analyzeRepository()`
- `generateCliView()`
- `summarizeReport()`
- `generateSlides()`

### 내부 표준화 항목
- provider
- model
- timeout
- retry
- max tokens
- response schema
- cost tracking
- latency logging

## 5. 운영 리스크와 대응

### 5.1 비용 리스크
LLM 호출량이 많아질수록 비용이 빠르게 증가할 수 있다.

대응:
- 기능별 기본 모델 지정
- 고급 모델 opt-in
- 캐싱
- 무료/유료 플랜 분리
- 재분석 제한

### 5.2 품질 리스크
분석 결과 품질이 일정하지 않으면 서비스 신뢰도가 떨어진다.

대응:
- structured output
- prompt versioning
- 골든 샘플 평가
- provider별 비교 테스트

### 5.3 보안 리스크
GitHub repo 분석에는 보안 이슈가 따른다.

체크 포인트:
- private repo 처리 정책
- OAuth 토큰 저장 방식
- 결과 접근 제어
- 외부 링크/첨부 검증
- 관리자 감사 로그

## 8. 최종 개발 원칙

Forkverse를 앞으로 발전시킬 때 아래 원칙을 유지해야 한다.

### 원칙 1
**분석 기능이 중심이고, 소셜은 그 결과를 유통하는 레이어다**

### 원칙 2
**공식 서비스가 중심이며, GitHub 공개는 유입/신뢰 전략이다**

### 원칙 3
**무거운 작업은 API와 분리해 Worker에서 처리한다**

### 원칙 4
**CLI 감성은 브랜드 표현이지만, 본문 가독성을 해치면 안 된다**

### 원칙 5
**기능 추가보다 제품 초점 정리가 먼저다**

## 9. 최종 결론

Forkverse는 단순한 개발자 SNS로 가기보다, 다음 방향으로 가야 가장 강하다.

### 최종 정의
**GitHub 저장소를 AI로 분석하고, 그 결과를 공유 가능한 개발자 인사이트로 바꿔주는 공식 SaaS 플랫폼**

### 최종 전략
- 제품: Repo Analysis Platform
- 운영: 중앙 DB 기반 SaaS
- 공개: 오픈코어
- UX: Analyze first, social second
- 기술: API + Worker + Job Queue + 확장 가능한 DB + LLM Gateway

즉, 앞으로의 모든 기획과 개발은 이 기준 위에서 정렬되어야 한다.

이 방향이 가장 제품성, 확장성, 공개 전략, 수익화 가능성, 개발자 유입 측면에서 균형이 좋다.
